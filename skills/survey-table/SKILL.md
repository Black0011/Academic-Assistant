# Survey Table — 论文调研总表与方向报告

## 何时使用

- 用户要求**调研单篇论文**时 → 按论文表格列定义输出分析，并追加到主表
- 用户要求**调研一个小方向**时 → 输出"论文表格 + 分类总结"的完整报告，并把论文追加到主表
- 用户要求**查看/更新调研总表**时 → 读取并维护 `data/survey/paper-table.md`

---

## 一、论文表格列定义（7 列）

对每篇论文，必须提取以下 7 个维度的信息。这是所有分析的最小单元。

| 列名 | 含义 | 填写要求 |
|------|------|---------|
| **类型** | 所属研究小方向 | 简短标签，如"生成式 Text generation"、"混合式 Table-to-Report"、"强化学习 过程推理"等。同一小方向的多篇论文共享同一标签 |
| **应用场景** | 论文聚焦的具体任务/场景 | 如 Text2SQL、data augmentation、过程推理等 |
| **公司/学校** | 第一作者或主要贡献机构 | 如"清华"、"微软"、"字节"、"波士顿大学" |
| **标题** | 论文完整标题 + 链接 | 格式：`[标题](URL)`，URL 优先使用论文官方链接或 arXiv 链接 |
| **核心内容** | 论文核心方法和思路 | **最重要的列**。用 3-8 条要点描述：解决什么问题、核心思路分几步、每步做什么。要让没读过论文的人能理解方法 |
| **优点** | 该方法的优势/借鉴点 | 2-4 条，聚焦方法创新性、实用性、效果提升。如果有具体数字请引用 |
| **缺点** | 该方法的局限/不足 | 2-4 条，关注适用范围限制、工程成本、数据依赖、未验证场景等 |

### 核心内容填写规范

核心内容是表格中信息量最大的一列，应当结构化书写：

```
**问题**：一句话说明论文要解决什么问题。
**方法**：核心思路概述（1-2 句）。
具体步骤：
1. 第一步做什么
2. 第二步做什么
3. 第三步做什么
**关键借鉴点**（可选）：对我们有什么直接参考价值。
```

## 二、主表维护规则

主表文件路径：`data/survey/paper-table.md`

### 主表格式（子表架构）

主表按研究方向拆分为多个子表，每个子表是一个独立的 Markdown 二级标题 + 表格。

```markdown
# 论文调研总表

> 最后更新：YYYY-MM-DD | 论文总数：N 篇 | 子表数：M

---

## A. {方向名称}

> 一句话描述该方向聚焦内容

| 类型 | 应用场景 | 公司/学校 | 标题 | 核心内容 | 优点 | 缺点 |
|------|---------|----------|------|---------|------|------|
| ... | ... | ... | ... | ... | ... | ... |

---

## B. {方向名称}

> 一句话描述

| 类型 | 应用场景 | 公司/学校 | 标题 | 核心内容 | 优点 | 缺点 |
|------|---------|----------|------|---------|------|------|
| ... | ... | ... | ... | ... | ... | ... |

---

*子表索引：A=xxx | B=xxx | ...*
```

### 当前子表分类

| 子表 | 方向 | 聚焦内容 |
|------|------|---------|
| A | 评估模型与方法 | 大模型评估系统、LLM-as-a-Judge、评测 Benchmark |
| B | 多任务强化学习 | 多任务 RL 算法、多任务 RLHF、多任务泛化方法 |
| C | 奖励模型与奖励塑形 | reward model 设计、LLM reward shaping、过程奖励、RLHF 基础 |
| D | 持续/终身学习 | continual RL、lifelong learning、灾难性遗忘 |
| E | 多任务 RL 环境与 Benchmark | 实验环境、数据集、评测基准 |

子表分类可随调研深入动态增减。新方向积累 3+ 篇论文时可新建子表。

### 维护原则

1. **按子表归类**：新论文追加到最匹配的子表中。子表内按"类型"列进一步分组
2. **去重**：追加前检查全部子表，标题不重复
3. **更新元信息**：每次追加后更新表头的"最后更新"日期、"论文总数"和"子表数"
4. **只收录优秀/有参考价值的论文**：不是所有读过的论文都要进主表
5. **跨方向论文**：如果一篇论文横跨多个方向，放入最核心的那个子表，在"类型"列标注交叉属性

### ⚠️ 安全写入规范（关键）

**`paper-table.md` 是跨会话积累的核心资产，必须遵守以下规则：**

1. **禁止 Write 覆盖整文件**：任何情况下都不得使用 `Write` 工具重写 `paper-table.md`
2. **必须先 Read 再追加**：每次修改前必须先读取当前文件，确认已有论文数量
3. **只能用 StrReplace 追加**：定位目标子表的最后一行数据，在其后插入新行
4. **新建子表时保留所有旧子表**：在文件末尾追加新子表，不改动已有内容
5. **如果操作后论文数量比操作前少，必须立即停止并告警**

```
❌ 绝对禁止：
   Write(path="data/survey/paper-table.md", contents="# 论文调研总表\n...")

✅ 正确做法：
   Read("data/survey/paper-table.md")  // 先读，确认有 N 篇
   StrReplace(定位子表末尾, 插入新行)  // 追加
   StrReplace(更新表头论文总数 N→N+k)  // 更新计数
```

## 三、单篇论文调研流程

当用户要求调研某一篇论文时：

### Step 1：阅读论文

使用 paper-reading skill 的三遍阅读法（至少完成第二遍），充分理解论文。

### Step 2：按 7 列提取信息

逐列填写，输出为 Markdown 表格的一行。注意：
- **核心内容**列必须条理清晰、信息充分
- **优点/缺点**必须是自己的判断，不能只复述摘要
- 有具体实验数据的要引用数字

### Step 3：输出格式

向用户展示单篇论文的分析结果，使用如下格式：

```markdown
### 论文分析：{论文简称}

| 类型 | 应用场景 | 公司/学校 | 标题 | 核心内容 | 优点 | 缺点 |
|------|---------|----------|------|---------|------|------|
| xxx | xxx | xxx | [标题](url) | 要点1；要点2；要点3 | 优点1；优点2 | 缺点1；缺点2 |
```

### Step 4：追加到主表

确认分析后，将该行追加到 `data/survey/paper-table.md` 主表对应类型区域。

## 四、小方向调研报告流程

当用户要求调研某个小方向时，输出报告应包含两大部分，与参考调研报告格式保持一致。

### 报告结构

```markdown
# {方向名称} 技术方案调研

## 一、技术方案调研

| 类型 | 应用场景 | 公司/学校 | 标题 | 核心内容 | 优点 | 缺点 |
|------|---------|----------|------|---------|------|------|
| ... | ... | ... | ... | ... | ... | ... |
（该方向下所有值得收录的论文，每篇一行）

## 二、调研总结

### 方案分类

按方法类型对上述论文进行归类，说明每类方案的：
- 核心思路
- 适用场景
- 代表性工作

（使用表格或分段落描述均可，参考格式如下）

| 方案类型 | 介绍 | 适用场景 |
|---------|------|---------|
| 类型A | 简要描述该类方法的共性思路 | 适合什么场景 |
| 类型B | ... | ... |

### 关键发现

按任务/场景维度总结现有评估/方法聚焦的主要对象，以及各自关注的重点指标。

### 综合分析

1-2 段综合性分析，指出：
- 当前最主流/最有前景的方案方向
- 各方案之间的互补关系
- 落地建议（成本、效果、工程复杂度的平衡）

### 研究空白与展望（可选）

如果调研深度足够，列出发现的研究空白和可能的探索方向。
```

### 报告写作原则

1. **论文表格在前，总结分析在后**——先给出事实，再给出判断
2. **总结必须基于表格中的论文**——不能凭空编造未读过的工作
3. **分类要有依据**——按方法类型、任务类型、或技术路线分类，选择最合适的维度
4. **综合分析要有观点**——不只是罗列，要给出"哪种方案更适合什么场景"的判断
5. **所有引用必须可追溯**——提到的论文必须出现在表格中

## 五、持续优化

本 Skill 随调研深入不断迭代。以下是预留的优化方向：

- **增加列**：如果发现 7 列不够用（例如需要"开源代码""数据集""关键指标"等列），可以在主表中扩展
- **增加评分维度**：为论文增加"推荐指数"或"与我们场景的匹配度"等主观评价列
- **跨方向对比**：当主表积累足够多方向时，支持跨方向的横向对比总结
- **自动关联**：与 paper-reading skill 的 YAML 笔记打通，自动从笔记生成表格行
